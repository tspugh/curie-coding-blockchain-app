# SPEC-0004.1: Temporary compliance repair ledger

Status: Temporary · Owner: tspugh · Date: 2026-06-01 · Delete when complete

> **Temporary fractional spec.** This file is an explicit violation ledger, not a
> durable roadmap spec. Fractional specs exist only to collect compliance gaps
> discovered between numbered specs. Every requirement below must be folded into
> its owning durable spec and implementation, then this file must be deleted.

## 1. Summary & User Story

This document lists the self-compliance failures found while reviewing branch
`spec-4-implementation` against SPEC-0001 through SPEC-0004. It exists to keep the
repair work visible while the durable specs and implementation are brought back
into alignment.

As a maintainer, I want one temporary list of known spec-compliance violations, so
that the next implementation pass can close them without pretending this
fractional spec is itself a permanent source of truth.

## 2. Requirements

- **R1 (MUST) Native Somnia execution path.** Real adjudication must originate
  from Somnia AI Agent Platform `createRequest` plus platform callback, not from
  `selfHosted` synthetic request ids or an off-chain Anthropic orchestrator. Owning
  durable spec: SPEC-0006, with downstream status cleanup in SPEC-0001,
  SPEC-0003, SPEC-0004, and SPEC-0005.
- **R2 (MUST) Coverage-arbiter payload.** The native-agent payload must include
  the coverage-arbitration inputs required by SPEC-0001 and SPEC-0004: request
  drug, quantity, requested amount, public evidence refs, attached policy URI,
  and policy criteria. A hard-coded website-number-extraction prompt is not
  compliant. Owning durable spec: SPEC-0006 for platform mechanics; SPEC-0001 and
  SPEC-0004 for payload semantics.
- **R3 (MUST) Frozen packet/store model.** Evidence packet flow must implement
  packet roots, packet URLs, `PacketSubmitted`, and a content-addressed Curie
  packet store keyed by Merkle root. Owning durable spec: SPEC-0004; verification
  gate: SPEC-0005.
- **R4 (MUST) Real arbiter reads formulary and packet bodies.** Real rulings must
  be grounded in pinned formulary releases, structured slices, and packet bodies;
  they must not judge only on-chain hashes and scalar amounts. Owning durable
  specs: SPEC-0004 and SPEC-0006.
- **R5 (MUST) Self-claim support.** Equal provider and payer addresses must work
  where SPEC-0001 requires single-wallet/self-claim mode, with party identity
  distinguished at the app layer. Owning durable spec: SPEC-0001; regression gate:
  SPEC-0005.
- **R6 (MUST) Appeal from any ruling state allowed by spec.** Appeals must follow
  SPEC-0001 state semantics, including approved rulings where either party may
  appeal. Tests must not assert narrower branch behavior as compliance. Owning
  durable spec: SPEC-0001.
- **R7 (MUST) Current policy-void semantics.** The normal on-label
  policy-void case must approve while flagging voided policy clauses; terminal
  `PolicyInvalidated` is reserved for the narrowed meta-policy failure described
  by current SPEC-0001/SPEC-0004 text. Owning durable specs: SPEC-0001,
  SPEC-0004, and SPEC-0006 decoder/schema details.
- **R8 (MUST) Transaction preview and estimate-before-send.** Real write paths
  must expose per-transaction cost previews and estimate/revert mapping before
  send. Owning durable spec: SPEC-0003; enforcement gate: SPEC-0005.
- **R9 (MUST) Pending guards on write buttons.** Every tx-firing UI action must
  disable and show pending state until the tx-confirmed path resolves. Owning
  durable spec: SPEC-0003; enforcement gate: SPEC-0005.
- **R10 (MUST) Create balance gate includes requested amount and gas.** Create
  must block when requested amount plus estimated create gas and reserve exceed
  balance. Owning durable spec: SPEC-0003; enforcement gate: SPEC-0005.
- **R11 (MUST) Custom-case and counterparty wallet flows.** The UI must expose
  editable custom-case data and per-instance provider/payer wallet configuration,
  not only current-wallet provider plus hard-coded payer. Owning durable spec:
  SPEC-0004; enforcement gate: SPEC-0005.
- **R12 (MUST) One-click policy-void gotcha.** Loading the non-compliant policy
  demo path must itself drive the gotcha outcome under the applicable current
  policy-void semantics; tests must not require a separate manual/test-only
  outcome selector to complete the set-piece. Owning durable spec: SPEC-0002,
  adjusted as needed for SPEC-0001/SPEC-0004 policy-void semantics.

## 3. Technical Documentation

This file does not define new architecture. Each requirement above must be routed
to one or more durable specs:

- SPEC-0001 owns protocol state-machine semantics: self-claim, appeals, policy
  flagging, and real native-agent ruling expectations.
- SPEC-0002 owns the demo set-piece and visible policy-void/gotcha experience.
- SPEC-0003 owns token-flow visibility, tx preview, pending guards, balance
  gating, and reasoning/receipt surfaces.
- SPEC-0004 owns data, evidence packet, formulary, custom-case, payer-line, and
  policy-rule semantics.
- SPEC-0005 owns end-to-end acceptance gates and regression scenarios.
- SPEC-0006 owns the Somnia AI Agent Platform integration and removal of
  self-hosted contract-execution pathways.

When a requirement is fixed, update the owning durable spec and its tests first.
Then mark the item closed in this file or delete the item. When no open items
remain, delete this entire file and remove its index entry.

## 5. Test Cases

- **T1 (R1-R4):** Real-mode adjudication produces a `Ruled` event through Somnia
  AI Agent Platform using the selected native agent, and the ruling is grounded in
  the submitted packet/formulary inputs.
- **T2 (R5-R7):** Protocol tests cover self-claim, appeal from all spec-allowed
  ruling states, and approve-with-policy-flag semantics.
- **T3 (R8-R10):** UI/real-backend tests prove cost preview, estimate-before-send,
  pending guards, and balance blocking before writes.
- **T4 (R11):** Browser scenario creates or loads a custom case with explicit
  provider and payer addresses, then verifies both wallets see and can act on the
  same request according to their roles.
- **T5 (R12):** Browser scenario loads the non-compliant policy gotcha and reaches
  the required policy-flag/void outcome without any hidden manual decision
  override.

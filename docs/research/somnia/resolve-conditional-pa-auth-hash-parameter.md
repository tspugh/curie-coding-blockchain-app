## 2026-05-17 — resolveConditionalPaStatus() paAuthHash parameter: YES, required for conditional→satisfied; PAS uses ClaimUpdate (not $inquire) to add clinical detail; authorizationNumber extension is item-level string on ClaimResponse

**Question answered:** Should `resolveConditionalPaStatus(bytes32 claimId, uint8 resolvedStatus)` also accept a `bytes32 paAuthHash` parameter — enabling `conditional → satisfied` (payer grants PA after receiving more clinical detail) in addition to `conditional → auth-needed` and `conditional → no-auth`?

---

### Findings

- **`Claim/$inquire` is a status-query operation only — it does NOT submit new clinical information.** The PAS IG OperationDefinition states that `$inquire` "is used to make a search for a Prior Authorization Response by example" following X12 278 Inquiry rules. It returns existing response state; it is not the mechanism for adding clinical detail to a pending PA request. The earlier research log entry stating "conditional resolves via PAS `$inquire`" was **incorrect** — `$inquire` was misidentified as the update pathway. ([PAS `$inquire` OperationDefinition](https://build.fhir.org/ig/HL7/davinci-pas/en/OperationDefinition-Claim-inquiry.html); [PAS artifacts](https://build.fhir.org/ig/HL7/davinci-pas/en/fhirArtifacts.html))

- **The correct pathway for `conditional → satisfied` is: hospital sends `ClaimUpdate` (not `$inquire`) → payer returns approved `ClaimUpdateResponse` with `extension:authorizationNumber`.** The PAS IG defines `ClaimUpdate` as the mechanism for "updating a previously submitted Claim instance" with additional information. The PAS Claim profile explicitly states: "Sequence numbers SHALL stay the same across all instances of the Prior Authorization, eg the Claim, ClaimResponse, ClaimUpdate, ClaimUpdateResponse." This is the normative follow-up path for pended/conditional claims. ([PAS Claim profile](https://build.fhir.org/ig/HL7/davinci-pas/StructureDefinition-profile-claim.html); [PAS IG](https://build.fhir.org/ig/HL7/davinci-pas/))

- **The PAS authorization number is `extension:authorizationNumber` on `ClaimResponse.item` — NOT the same field as the CRD `satisfied-pa-id`.** Two distinct extensions exist in the PAS ClaimResponse profile: (1) `extension:authorizationNumber` — "A string assigned by the UMO to an authorized review outcome associated with this service item" — this is the **approval identifier** returned by the payer when PA is granted; (2) `extension:previousAuthorizationNumber` — references an already-known prior auth number from a different earlier authorization context, not the outcome identifier for the current approval. The `conditional → satisfied` resolution produces a `ClaimUpdateResponse` carrying `extension:authorizationNumber`, not `previousAuthorizationNumber`. ([PAS ClaimResponse profile](https://build.fhir.org/ig/HL7/davinci-pas/StructureDefinition-profile-claimresponse.html); [authorizationNumber extension](https://build.fhir.org/ig/HL7/davinci-pas/branches/__default/en/StructureDefinition-extension-authorizationNumber.html))

- **`conditional → satisfied` is a valid PAS workflow path but is not normatively prescribed as an explicit end-to-end sequence in either the CRD or PAS IG.** The IGs separate concerns: CRD discovers requirements (returns `conditional`); DTR collects additional clinical documentation; PAS handles the formal PA submission and response. The full sequence — CRD `conditional` → DTR data collection → PAS `ClaimUpdate` → approved `ClaimUpdateResponse` with `authorizationNumber` — is workflow-consistent with the IGs but is an implied integration pattern rather than a single normative mandate. (Weakly sourced: derived from IG architecture; no single canonical sequence diagram covers this path.)

- **`authorizationNumber` is a `string` (same underlying type as CRD `satisfied-pa-id`), and the same hash derivation applies.** `PA_HASH_DOMAIN_SEPARATOR = keccak256("cliqueue.v1.paAuthHash")`, then `paAuthHash = keccak256(abi.encodePacked(PA_HASH_DOMAIN_SEPARATOR, authorizationNumber))`. The hash computation is identical regardless of whether the authorization number came from CRD's `satisfied-pa-id` (initial PA check) or PAS's `extension:authorizationNumber` (conditional → satisfied resolution). ([PAS authorizationNumber extension](https://build.fhir.org/ig/HL7/davinci-pas/branches/__default/en/StructureDefinition-extension-authorizationNumber.html))

- **Valid terminal resolutions of `conditional` are four, not two.** Prior research identified only `auth-needed` and `no-auth` as valid resolutions. Adding the `ClaimUpdate → approved` path means `satisfied` (0x02) is also a valid resolution of `conditional`. All valid resolutions: `no-auth (0x01)`, `satisfied (0x02)`, `auth-needed (0x03)`. The `performpa` (0x04) and `conditional` (0x05) values are not valid resolutions (the first belongs to a different CRD code meaning, the second would leave the claim in the same state).

- **Prior research log entry for `ClaimPaStatusResolved` stated an incorrect constraint.** The `resolveConditionalPaStatus` function was specified with `require(resolvedStatus == 0x03 || resolvedStatus == 0x01)` — excluding `satisfied`. This must be corrected to `require(resolvedStatus == 0x01 || resolvedStatus == 0x02 || resolvedStatus == 0x03)` and the function signature must accept an optional `bytes32 paAuthHash` validated conditionally.

- **Correct Solidity function signature:** `resolveConditionalPaStatus(bytes32 claimId, uint8 resolvedStatus, bytes32 paAuthHash) external onlyRole(HOSPITAL_AGENT_ROLE)` with guard: `require(resolvedStatus == 0x01 || resolvedStatus == 0x02 || resolvedStatus == 0x03, "Invalid resolution"); require(resolvedStatus == 0x02 ? paAuthHash != bytes32(0) : paAuthHash == bytes32(0), "paAuthHash mismatch")`. Emit `ClaimPaStatusResolved(claimId, resolvedStatus, block.timestamp)` — the `paAuthHash` does NOT need to be re-emitted in this event (it was already emitted in `ClaimSubmitted` for the satisfied case, or can be emitted as non-indexed data if auditability requires it).

- **Gas consideration:** Adding `bytes32 paAuthHash` as a function parameter costs no additional event gas. The `ClaimPaStatusResolved` event remains LOG3 + 32-byte data = 23,680 gas. If `paAuthHash` is also emitted as a non-indexed event field (for completeness when resolvedStatus = 0x02), the event becomes LOG3 + 64-byte data = `3200 + 15360 + 160×64 = 28,820 gas` — 5,160 gas more, negligible.

---

**Design implication:** Correct the `resolveConditionalPaStatus()` spec: (1) add `bytes32 paAuthHash` parameter, (2) expand valid resolutions to include `0x02 (satisfied)`, (3) enforce `paAuthHash != bytes32(0) iff resolvedStatus == 0x02`, (4) correct the prior log entry's statement that `$inquire` is the conditional resolution pathway — it is `ClaimUpdate`; and optionally emit `paAuthHash` as non-indexed data in `ClaimPaStatusResolved` when resolvedStatus = 0x02 for full on-chain auditability.

**Open questions generated:**
1. Should `ClaimPaStatusResolved` emit `paAuthHash` as a non-indexed `bytes32` data field when `resolvedStatus = 0x02 (satisfied)` — adding 32 bytes and 5,120 gas — so the authorization number commitment is auditable in the resolution event itself, rather than requiring auditors to correlate across `ClaimSubmitted` (where paAuthHash was zero for the original conditional submission)?
2. Should the hospital BAA's "PA Conditional Resolution SLA" clause specify different time windows for `conditional → auth-needed` (shorter, no further PA effort needed) vs. `conditional → satisfied` (may require DTR documentation collection + ClaimUpdate round-trip, longer window appropriate)?
3. Should cliqueue's `ClaimPaLifecycle` TypeScript module export a `resolveConditionalStatus(claimId, resolvedStatus, authorizationNumber?)` helper that computes `paAuthHash` from `authorizationNumber` and validates the pairing — encapsulating both the CRD `satisfied-pa-id` path and the PAS `extension:authorizationNumber` path in a unified function?

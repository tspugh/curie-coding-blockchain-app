## 2026-05-17 — PaStatusEncoder TypeScript module: no Da Vinci/CRD npm primitives exist; full hand-roll required; viem encodePacked is correct bytes32 derivation approach

**Question answered:** Does any production-ready TypeScript npm package provide Da Vinci CRD paStatus typed constants, authorizationNumber-to-bytes32 hashing helpers, or PAS ClaimUpdate workflow types that cliqueue could build on — or must `PaStatusEncoder` / `ClaimPaLifecycle` be hand-rolled entirely?

- **No Da Vinci CRD/PAS TypeScript npm package exists as of May 2026.** The npm registry contains no package named `davinci-crd`, `davinci-pas`, `hl7-davinci`, or any equivalent. HL7's `github.com/HL7/davinci-crd` and `github.com/HL7-Davinci/CRD` are IG source repos only — no published npm artifact. ([HL7/davinci-crd GitHub](https://github.com/HL7/davinci-crd/); [HL7-Davinci/CRD GitHub](https://github.com/HL7-Davinci/CRD))
- **Generic FHIR TypeScript packages (`@medplum/fhirtypes`, `@types/fhir`, `@ahryman40k/ts-fhir-types`) do not export Da Vinci extension URLs or `coveragePaDetail` ValueSet codes.** `@medplum/fhirtypes` is the most complete general-purpose FHIR type package but its codegen is centered on core FHIR resources; it does not expose `http://hl7.org/fhir/us/davinci-crd/ValueSet/coveragePaDetail` as typed constants. ([Medplum fhirtypes on GitHub](https://github.com/medplum/medplum/tree/main/packages/fhirtypes); [npm: @types/fhir](https://www.npmjs.com/package/@types/fhir))
- **No CDS Hooks TypeScript client (confirmed absent last iteration) ships CRD Coverage Information extension types.** The gap identified in [[../regulatory/cds-hooks-typescript-client-crd-v2-1|cds-hooks-typescript-client-crd-v2-1]] extends to the extension type layer — even if a CDS Hooks client existed, it would not provide CRD-specific extension structs.
- **Correct viem derivation for `paAuthHash`:** `keccak256(encodePacked(['bytes32', 'string'], [PA_HASH_DOMAIN_SEPARATOR, authorizationNumber]))`. Uses viem's `encodePacked` + `keccak256`, which exactly mirrors Solidity `keccak256(abi.encodePacked(bytes32, string))`. Ethers v6 equivalent: `solidityPackedKeccak256(['bytes32', 'string'], [PA_HASH_DOMAIN_SEPARATOR, authorizationNumber])`. ([viem encodePacked discussion](https://github.com/wevm/viem/discussions/596); [ethers v5 packed hashing docs](https://docs.ethers.org/v5/api/utils/hashing/))
- **`PaStatusEncoder` module must be entirely hand-rolled.** Required exports: `enum PaStatus { NOT_CHECKED = 0x00, NO_AUTH = 0x01, SATISFIED = 0x02, AUTH_NEEDED = 0x03, PERFORMPA = 0x04, CONDITIONAL = 0x05 }`, `PA_HASH_DOMAIN_SEPARATOR: bytes32` (= `keccak256("cliqueue.v1.paAuthHash")`), `derivePaAuthHash(authorizationNumber: string): Hex`, `validatePaAuthHash(paStatus: PaStatus, paAuthHash: Hex): void` (enforces zero/non-zero invariant). All four exports are unique to cliqueue's on-chain design.
- **`ClaimPaLifecycle` module wraps contract interactions:** `resolveConditionalStatus(claimId: Hex, resolvedStatus: PaStatus, authorizationNumber?: string): Promise<TransactionReceipt>` — handles both the `no-auth`/`auth-needed` path (paAuthHash = bytes32(0)) and the `satisfied` path (derives paAuthHash from authorizationNumber). No library provides this; it must be spec'd as a TypeScript module alongside `PaStatusEncoder`.
- **Publishing as `@cliqueue/pa-lifecycle` fills a real ecosystem gap.** The absence of any Da Vinci CRD TypeScript primitives on npm means an open-source `@cliqueue/pa-lifecycle` package would be the first published typed CRD paStatus enum + PAS lifecycle helper in the TypeScript ecosystem — analogous to the `@cliqueue/cds-hooks-client` gap identified last iteration.

**Design implication:** `PaStatusEncoder` and `ClaimPaLifecycle` must be hand-rolled as a single `@cliqueue/pa-lifecycle` TypeScript package. The viem `encodePacked(['bytes32','string'], ...)` pattern is confirmed correct for `paAuthHash` derivation. Publishing this as open-source fills an ecosystem gap that strengthens hospital onboarding (integration engineers can import typed constants rather than hardcoding URL strings) and creates a defensible go-to-market collateral artifact alongside the `@cliqueue/cds-hooks-client` gap.

**Open questions generated:**
1. Should `@cliqueue/pa-lifecycle` and `@cliqueue/cds-hooks-client` be published as a monorepo (`@cliqueue/*`) with shared FHIR extension URL constants — preventing URL string drift between the CDS Hooks client and the PA lifecycle encoder?
2. Should `@cliqueue/pa-lifecycle` include a `fromCoverageInformationTask(task: fhir4.Task): PaStatusResult` parser that extracts `paStatus` and `satisfied-pa-id` from a CDS Hooks `systemActions[].resource` Task — bridging the CDS Hooks client output to the on-chain encoder input?
3. Should the `@medplum/fhirtypes` package be evaluated as a `peerDependency` of `@cliqueue/pa-lifecycle` for typing FHIR Task and Coverage resources — or should cliqueue vendor only the minimal extension interfaces needed to avoid pulling in the full Medplum type tree?

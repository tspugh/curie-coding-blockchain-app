## 2026-05-17 — Reverse proxy / mTLS patterns for MCP server authentication without URL-embedded API keys — feasible via nginx credential injection; mTLS also solves the problem but requires client-cert capability; Somnia `inferToolsChat` is permanently constrained to unauthenticated or URL-keyed MCP endpoints

**Question answered:** Is there a reverse proxy or mTLS pattern for securing MCP server access without URL-embedded API keys — and does any such pattern work within Somnia's `inferToolsChat` `mcpServerUrls` constraint (which accepts only a `string[]` URL array, no headers or credential parameters)?

**Key findings:**

- **Nginx reverse proxy credential injection is architecturally sound and widely deployed.** Nginx (and any HTTPS reverse proxy) can accept inbound requests from `inferToolsChat` with no credentials in the URL, and independently inject the real API key to the upstream MCP server via `proxy_set_header Authorization "Bearer <key>"` or `proxy_pass https://upstream?key=<key>`. The key lives in the nginx server configuration — never in the `mcpServerUrls` array, never in Somnia calldata. This is the canonical "API key on the front-end" protection pattern. ([Jeremy Poole — Protecting API keys on the front-end using NGINX](https://jeremypoole.ca/posts/protecting_api_keys_on_the_front_end/); [fast.io — MCP Server Proxy Setup](https://fast.io/resources/mcp-server-proxy/))

- **mTLS between nginx and the upstream MCP server eliminates secret credentials entirely.** Nginx supports `proxy_ssl_certificate` / `proxy_ssl_certificate_key` directives that cause nginx to present a client certificate to the upstream MCP server during the TLS handshake — with zero API keys anywhere. The downstream connection (Somnia `inferToolsChat` → nginx) can be plain HTTPS with no client cert required. This asymmetric pattern is directly supported by NGINX's upstream TLS documentation. ([NGINX — Securing HTTP Traffic to Upstream Servers](https://docs.nginx.com/nginx/admin-guide/security-controls/securing-http-traffic-upstream/); [DEV Community — MCP Authentication: OAuth vs API Keys vs Mutual TLS](https://dev.to/whoffagents/mcp-server-authentication-oauth-vs-api-keys-vs-mutual-tls-which-to-use-and-when-4nj3))

- **mTLS does NOT solve the Somnia `inferToolsChat` → nginx leg.** mTLS requires the *client* to present a certificate. Somnia's `inferToolsChat` can only pass a URL string — it cannot present a TLS client certificate. mTLS therefore cannot authenticate the connection from Somnia validators to the nginx proxy. mTLS is useful exclusively on the nginx-to-upstream (nginx-to-Corti Symphony) leg, not the Somnia-to-nginx leg. (DEV Community source above, citing "mutual TLS flips the standard TLS handshake...the client must present a valid certificate — this is non-negotiable")

- **OAuth 2.1 and bearer token patterns also require header injection from the client — not available in `inferToolsChat`.** The MCP November 2025 spec mandates OAuth 2.1 + PKCE for public remote servers. All standard MCP authentication patterns (OAuth, Bearer, JWT, API-Management subscription keys) require the client to send credentials in HTTP headers (`Authorization`, `Ocp-Apim-Subscription-Key`, etc.). Somnia `inferToolsChat` passes only URL strings — no client-side header injection capability. Azure API Management's "secure inbound access" pattern and FastMCP OAuth proxy both assume the MCP client can set headers. ([Microsoft Learn — Secure access to MCP servers in Azure API Management](https://learn.microsoft.com/en-us/azure/api-management/secure-mcp-servers); [FastMCP — Authentication](https://gofastmcp.com/servers/auth/authentication))

- **The confirmed feasible pattern for cliqueue: hospital-operated nginx proxy accepting unauthenticated inbound from `inferToolsChat`, injecting Corti Symphony API key outbound.** The nginx proxy is deployed as part of the hospital's HIPAA-compliant off-chain infrastructure (co-located with the SQS/S3 claim package stack). The `mcpServerUrls` entry points to the hospital's nginx proxy FQDN (`https://<hospital-proxy>.hospital-domain.internal/mcp`). Nginx holds the Corti API key in server config (encrypted at rest, never on-chain). The upstream call is either: (A) nginx injects `Authorization: Bearer <corti-key>` header, or (B) nginx → Corti via mTLS (no key at all, mutual cert auth). Option B (mTLS) is operationally more complex but eliminates the long-lived secret entirely.

- **Static egress IP for hospital proxy solves the Somnia validator → hospital allowlisting problem.** Since Somnia validators have no published static egress IPs (confirmed in prior iteration), the hospital-proxy approach inverts the trust model: instead of allowlisting Somnia validator IPs at the hospital MCP endpoint, the hospital deploys the proxy with a **static public IP** (via services like QuotaGuard Shield, or a fixed cloud EIP/NAT gateway). The proxy validates that incoming requests are structurally valid MCP JSON before proxying — no IP allowlisting of Somnia validators required. ([QuotaGuard — Give Your MCP Server a Static Egress IP](https://www.quotaguard.com/blog/mcp-server-static-egress-ip)) WEAK: QuotaGuard is for *outbound* egress from the hospital side; in cliqueue's case the Somnia validator is the client calling *into* the hospital proxy, which is an *inbound* allowlisting problem, not outbound. The relevant control is the hospital proxy's inbound IP allowlist, which remains infeasible without Somnia static egress IPs.

- **CONFIRMED GAP: No IP-based inbound allowlist is feasible for the Somnia → hospital proxy call.** If the hospital nginx proxy has no authentication on the inbound leg (Somnia calls it unauthenticated), any party on the internet who discovers the proxy URL can call it and trigger credential injection to Corti. This is the residual risk of the nginx credential-injection pattern: the URL itself becomes the secret. URL-as-secret is acceptable only if the URL is a sufficiently long random identifier (128-bit entropy path segment) and is treated as a credential (rotated periodically, not committed to public repos, not emitted in on-chain events). (WEAK — no primary source; standard web security posture)

- **Conclusion: The nginx credential-injection pattern is viable for cliqueue's `inferToolsChat` use case, with the URL path segment as the sole inbound authenticator.** The architecture is: (1) hospital deploys nginx proxy at `https://<fqdn>/<128-bit-random-path>/mcp`, (2) `mcpServerUrls` in on-chain `inferToolsChat` call contains this URL (the path segment is the secret), (3) nginx validates the path, injects Corti API key, proxies the MCP call, (4) the `<128-bit-random-path>` is never logged on-chain and is treated as a rotatable credential in the hospital's secret management system, (5) the random path segment is revocable (swap to a new path without changing contracts — `mcpServerUrls` is a parameter, not a storage slot). This pattern is architecturally weaker than OAuth/mTLS from the hospital inbound leg but is the only option compatible with `inferToolsChat`'s URL-only constraint.

**Design implication:** cliqueue's off-chain agent spec must document the hospital nginx proxy pattern as a normative architecture for `inferToolsChat` calls that reach authenticated external MCP servers: (A) hospital deploys nginx with 128-bit random path segment as inbound authenticator, (B) nginx injects Corti API key outbound (or uses mTLS on the nginx→Corti leg), (C) the random path is a rotatable secret managed in the hospital's secret management system (never emitted on-chain, never logged), (D) the hospital BAA Off-Chain Infrastructure Exhibit must include the nginx proxy as a named component with its security posture documented. The `mcpServerUrls` array in `inferToolsChat` is confirmed to be URL-as-secret — this is the only available inbound mechanism given Somnia's no-header constraint.

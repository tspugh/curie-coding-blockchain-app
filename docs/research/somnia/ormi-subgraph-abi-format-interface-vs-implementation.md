## 2026-05-17 — Ormi subgraph ABI format: interface ABI confirmed correct for ClaimsAdjudicator; clean JSON array required (not forge build JSON); custom error types safe in current graph-node; implementation ABI for UUPS proxy pattern

**Question answered:** Should the Ormi subgraph manifest use the `IClaimsAdjudicator` interface ABI (minimal, events + functions declared in the interface) or the Foundry implementation ABI (full, including storage layout artifacts from `forge build`) — and does the Foundry implementation ABI contain any artifacts that break `graph codegen` type generation?

- **The ABI supplied to `subgraph.yaml` must be a clean standalone JSON array `[...]`**, NOT the full Foundry `forge build` output JSON (which wraps the ABI array inside a top-level object with `abi`, `bytecode`, `deployedBytecode`, `storageLayout`, `ast`, `metadata` keys). Supplying the full forge build JSON as an ABI file to graph-node causes a deserialization failure. Canonical workaround: `forge inspect ClaimsAdjudicator abi > abis/IClaimsAdjudicator.json` or `jq '.abi' out/ClaimsAdjudicator.sol/ClaimsAdjudicator.json > abis/IClaimsAdjudicator.json`. ([Foundry forge inspect docs](https://book.getfoundry.sh/reference/forge/forge-inspect); [The Graph subgraph manifest](https://thegraph.com/docs/en/subgraphs/developing/creating/subgraph-manifest/))

- **The interface ABI (`IClaimsAdjudicator` + `IClaimsEvents`) is the correct artifact for the ClaimsAdjudicator data source in `subgraph.yaml`.** Ormi (and graph-node) index events present in the ABI file bound to the data source. Since prior research confirmed `IClaimsEvents.sol` declares all four indexable events (`ClaimSubmitted`, `ClaimAdjudicated`, `PayerClaimRefSet`, `ClaimPaStatusResolved`), the interface ABI is sufficient and preferred — it is smaller, contains no storage-layout artifacts, and avoids surfacing internal function selectors to external tooling. ([The Graph subgraph manifest — abis field](https://thegraph.com/docs/en/subgraphs/developing/creating/subgraph-manifest/))

- **For UUPS proxy contracts, the subgraph uses the proxy contract address in `subgraph.yaml` but the implementation contract's ABI in `abis/`.** This is The Graph's canonical proxy pattern: graph-node listens at the proxy address for events that match the implementation ABI's event signatures. The ABI must include all events that the implementation emits. For `ClaimsAdjudicator` (UUPS proxy), the correct ABI artifact is the compiled `IClaimsAdjudicator` interface ABI (which includes `IClaimsEvents` events via inheritance), NOT the raw proxy contract ABI (which contains only `IERC1967` upgrade events). ([Using The Graph for indexing a proxy contract](https://medium.com/@bsse1202/using-the-graph-for-indexing-a-proxy-contract-91ed48e2443e); [Chainstack UUPS subgraph guide](https://docs.chainstack.com/docs/creating-a-subgraph-for-upgradeable-proxy-contracts-a-developers-guide))

- **Ormi is fully The Graph-compatible.** Ormi deploys via the standard `graph deploy --node https://api.subgraph.somnia.network/deploy --ipfs https://api.subgraph.somnia.network/ipfs` CLI. ABI auto-fetch is supported from Somnia's block explorer; manual upload is the fallback. No Ormi-specific ABI format deviation from The Graph spec. ([Ormi Subgraph — Somnia Docs](https://docs.somnia.network/developer/building-dapps/data-indexing-and-querying/ormi-subgraph); [Ormi no-code subgraph guide](https://blog.ormilabs.com/how-to-deploy-a-no-code-subgraph-in-3-minutes-real-time-blockchain-indexing-with-ormi/))

- **Solidity custom error types (`type: "error"`) in the ABI previously broke graph-node deployment** with `"Serialization error: Invalid operation type error"`. This was fixed in graph-node in January 2022 (PR #3144, ethabi dependency update). Current graph-node and Ormi versions accept ABIs containing custom error entries. However, the recommended practice is still to use the interface ABI (`IClaimsAdjudicator`), which does not declare custom errors — leaving them only in the implementation and ensuring the subgraph ABI stays minimal. ([graph-node issue #2577](https://github.com/graphprotocol/graph-node/issues/2577); [graph-node PR #3144](https://github.com/graphprotocol/graph-node/pull/3144))

- **Uniswap v2-subgraph uses clean JSON ABI arrays** (standalone `[...]` arrays, no bytecode/storageLayout). The Uniswap v2 `pair.json` ABI contains events, functions, and a constructor — no custom errors, no top-level forge build wrapper. This is the DeFi-canonical pattern: extract ABI arrays from forge output and commit them as separate files in `abis/`. ([Uniswap v2-subgraph abis/pair.json](https://raw.githubusercontent.com/Uniswap/v2-subgraph/main/abis/pair.json))

**Design implication:** The Ormi subgraph `subgraph.yaml` for `ClaimsAdjudicator` uses the proxy contract address with the `IClaimsAdjudicator` interface ABI (clean JSON array extracted via `forge inspect IClaimsAdjudicator abi`). The second data source for `ISBTRegistry` uses the same pattern. The Foundry contracts repo CI should run `forge inspect <ContractName> abi > abis/<ContractName>.json` as part of the wagmi codegen step, and these clean ABI files are committed alongside `src/generated.ts`.

# SBTRegistry Attestor Count: Explicit Mapping vs. AccessControlEnumerable getRoleMemberCount

## 2026-05-16 — Should `SBTRegistry` maintain an explicit `mapping(bytes32 hospitalId => uint32) private _attestorCount` to make the floor check O(1), or derive it via `getRoleMemberCount` on a per-hospital role bytes32 — and what are the Somnia gas cost implications?

- **`getRoleMemberCount` reads exactly 1 storage slot** — it calls `_roleMembers[role].length()` which returns `set._values.length`, stored at a single slot derived from the EnumerableSet struct. There is no iteration. The function itself is O(1) on-chain. ([OpenZeppelin EnumerableSet.sol](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/utils/structs/EnumerableSet.sol); [AccessControlEnumerable.sol](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/access/extensions/AccessControlEnumerable.sol))
- **`AccessControlEnumerable.grantRole` costs 2 extra SSTOREs vs. base `AccessControl`** — the `EnumerableSet._add` function writes: (1) appends the new address to `_values[]` (one SSTORE for array element + one SSTORE updating array length, costed as a single `SSTORE` for the element at `values[len]` and an implicit length extension), (2) writes `_positions[value] = len + 1` (one SSTORE to the positions mapping). This is 2 additional SSTOREs per `grantRole` and per `revokeRole`, on top of the base `_roles[role].members[account] = true` SSTORE. ([OZ PR #2512: Remove enumerable from AccessControl](https://github.com/OpenZeppelin/openzeppelin-contracts/pull/2512); OZ community: "granting and revoking roles will be a lot more expensive with enumerability")
- **Somnia cold SLOAD = 1,000,000 gas** for any slot not in the IceDB 128M-slot LRU cache; **warm SLOAD = 100 gas** (static op fee only). The LRU cache is Somnia's core gas innovation — slots written or read recently are ~10,000× cheaper to read than cold slots. ([Somnia Docs: Gas Differences](https://docs.somnia.network/developer/somnia-gas-differences-to-ethereum))
- **Cold SLOAD risk for `getRoleMemberCount` at claim submission time is REAL**: SBT minting (`grantRole`) occurs at onboarding — days or weeks before first claim submission. By claim time, the `_roleMembers[hospitalRoleHash]._values.length` slot is highly likely to have been evicted from the 128M LRU cache. A 1M-gas cold SLOAD is triggered at every claim submission that checks the minimum attestors floor via `getRoleMemberCount`. At 10,000 claims/day, this is 10,000 × 1M gas cold reads per day if no warm-up mechanism exists.
- **An explicit `mapping(bytes32 hospitalId => uint32) private _attestorCount`** eliminates the EnumerableSet slot dependency for the floor check. The same cold-SLOAD risk still applies to the counter mapping slot itself — but it is a single dedicated slot that can be managed independently. More importantly, the counter slot is written on every `grantRole`/`revokeRole`, so it has a much higher probability of being warm in the IceDB cache if attestor management is frequent. With an explicit counter, `ClaimsAdjudicator` reads only `_attestorCount[hospitalId]` (1 SLOAD) rather than traversing the EnumerableSet path (1 SLOAD for `._values.length`, but also 1–2 additional SLOADs for EnumerableSet struct layout indirection on Somnia's ERC-7201 namespaced storage).
- **The explicit counter approach also reduces `grantRole`/`revokeRole` write overhead**: 1 SSTORE for `_attestorCount[hospitalId]++` vs. 2 SSTOREs for EnumerableSet `_add`/`_remove`. For a network-scoped `SBTRegistry` with many hospitals, the savings accumulate.
- **`AccessControlEnumerable` is NOT required for the floor check** — the enumeration capability (`getRoleMember(role, index)`) is only needed for off-chain queries (audits, dashboards). Enumeration can be tracked off-chain via `RoleGranted`/`RoleRevoked` events using The Graph or Ormi indexer without any on-chain enumerable storage. Using base `AccessControl` + explicit `_attestorCount` mapping eliminates the enumerable overhead entirely while preserving the count check on-chain.
- **Recommended architecture**: Use `AccessControl` (not `AccessControlEnumerable`) in `SBTRegistry`. Maintain `mapping(bytes32 hospitalId => uint32) private _attestorCount` updated in `_grantRole` / `_revokeRole` overrides. Emit `RoleGranted` / `RoleRevoked` events (already emitted by `AccessControl`) for off-chain enumeration via Ormi. This gives O(1) on-chain floor check with the smallest possible cold-SLOAD exposure. ([Somnia IceDB docs](https://docs.somnia.network/concepts/somnia-blockchain/somnias-icedb); [OZ issue: "Enumerable can be tracked offchain using tools like thegraph"](https://github.com/openzeppelin/openzeppelin-contracts/issues/5821))

**Design implication:** `SBTRegistry` should use base `AccessControl` (not `AccessControlEnumerable`) and maintain an explicit `mapping(bytes32 hospitalId => uint32) private _attestorCount`. The floor check in `ClaimsAdjudicator` reads this single slot. Off-chain enumeration of attestors (for audit dashboards) is handled via `RoleGranted`/`RoleRevoked` event indexing in Ormi — zero gas overhead. This is the O(1) gas path and eliminates 2 extra SSTOREs per credential operation. On Somnia, the cold-SLOAD risk for the counter slot is the same as for any other slot, but the counter slot has higher write frequency (updated at every credential change) than the EnumerableSet `_values.length` slot (only updated at credential change, same frequency — tie broken by the additional EnumerableSet structural overhead).

**Open questions generated:**
1. Should the `_attestorCount` mapping use `bytes32 hospitalId` keys (consistent with the `CLIQUEUE_DOMAIN_SEPARATOR` pattern) or `address hospitalAdmin` keys — and does mixing key types between `AccessControl` role hashes and the counter mapping create a key-derivation footgun?
2. Should `SBTRegistry` emit a dedicated `AttestorCountChanged(bytes32 hospitalId, uint32 newCount, bool isGrant)` event on every `_grantRole`/`_revokeRole` — so off-chain indexers have a single authoritative event for count tracking without reconstructing from `RoleGranted`/`RoleRevoked` pairs?
3. At what `_attestorCount` slot write frequency (credentials per hospital per month) does the counter slot reliably stay within Somnia's 128M LRU cache — and is there a minimum "keep-warm" operation (e.g., a monthly attestor verification ping) that cliqueue should specify in the deployment runbook?

## 2026-05-17 — CDS Hooks TypeScript client for CRD v2.1.0: no production npm package exists; hand-rolled fetch client is the correct approach; paStatus has 5 values not 4

**Question investigated:** For cliqueue's TypeScript hospital agent implementing the CRD CDS Hooks pre-submission prior-auth check: is there a production-ready `@cds-hooks/client` npm package or TypeScript CDS Hooks client library that supports the `order-sign` hook with the Coverage Information system action extension required by Da Vinci CRD IG v2.1.0 — or must cliqueue implement the CDS Hooks client from scratch?

### Key findings

- **No production-ready TypeScript CDS Hooks client npm package exists as of May 2026.** The npm registry contains only: `cds-hooks` (abandoned 2020, ISC, neuquen.gov.ar), `@asteen3/cds-hooks` (2022, Georgia Tech research), `@asymmetrik/node-cds-hooks` (2020, server-side only), and `@molit/cds-card-viewer` (2025, React UI component for displaying cards — not a request client). There is no `@cds-hooks/client`, `cds-hooks-client`, or `@types/cds-hooks` package. No DefinitelyTyped entry exists. ([npm search: cds-hooks](https://www.npmjs.com/search?q=cds-hooks); [npm registry API](https://registry.npmjs.org/-/v1/search?text=cds-hooks))

- **The HL7 `github.com/cds-hooks` org provides only server-side tooling and sandbox demos.** The `cds-hooks/sandbox-cds-services` repo is a Node.js server. The `HL7-DaVinci/crd-request-generator` is a JavaScript (not TypeScript) standalone web UI (React/Node, webpack), maintained by Lantana Consulting Group — not an importable npm package. The HL7 `github.com/HL7-DaVinci/CRD` reference implementation is primarily Java (80.2%) with JavaScript UI only. No TypeScript client SDK exists in these orgs. ([github.com/cds-hooks/sandbox-cds-services](https://github.com/cds-hooks/sandbox-cds-services); [github.com/HL7-DaVinci/crd-request-generator](https://github.com/HL7-DaVinci/crd-request-generator))

- **The CDS Hooks 2.0 wire protocol is simple enough to implement with plain `fetch` + TypeScript interfaces — no SDK is needed.** The protocol is a `POST` to `{baseUrl}/cds-services/{service.id}` with a JSON body containing: `hook` (required string), `hookInstance` (required UUID), `fhirServer` (conditional URL), `context` (required hook-specific object), `fhirAuthorization` (conditional Bearer token object), and `prefetch` (optional FHIR resource map). The response is a `{ cards: Card[], systemActions: Action[] }` object. ([CDS Hooks 2.0 specification](https://cds-hooks.hl7.org/2.0/))

- **The payer's CRD service discovery endpoint (`GET /cds-services`) returns a `prefetch` template that tells the client which FHIR resources to pre-fetch.** The client must call discovery first, read the service's `prefetch` object (FHIR query strings with `{{context.*}}` template variables), resolve those queries against the hospital's FHIR server, and include the results in the `prefetch` field of the hook call. The client must NOT hardcode a fixed set of resources; different payer CRD endpoints will declare different prefetch requirements. ([CDS Hooks 2.0 spec: prefetch](https://cds-hooks.hl7.org/2.0/); [CDS Hooks cheat sheet](https://cds-hooks.org/cheat-sheet/cheat-sheet/))

- **The CRD `order-sign` hook context must include `userId`, `patientId`, `encounterId` (optional), and `draftOrders` (the FHIR Bundle of orders being signed).** At order-sign, all draft orders are considered "complete" — the payer expects final order data, not provisional. The CRD IG also permits re-triggering order-sign when orders are updated (status change, timeframe change). The CRD IG requires CRD clients to invoke all payer CRD endpoints in parallel for multi-coverage scenarios, not sequentially. ([Da Vinci CRD STU2.1 deviations](https://hl7.org/fhir/us/davinci-crd/STU2.1/deviations.html); [CDS Hooks order-sign](https://cds-hooks.org/hooks/order-sign/))

- **The `coveragePaDetail` ValueSet in CRD STU2.1 defines FIVE paStatus codes — not four as previously documented in this repo.** The correct complete list: ([CRD STU2.1 ValueSet/coveragePaDetail](https://hl7.org/fhir/us/davinci-crd/STU2.1/ValueSet-coveragePaDetail.html))
  - `no-auth` — ordered service does not require prior authorization
  - `auth-needed` — ordered service will require prior authorization
  - `satisfied` — conditions evaluated by prior authorization already satisfied; PA can be bypassed
  - `performpa` — PA needed but must be initiated by the performing (not ordering) provider
  - `conditional` — cannot determine if additional info requirements exist without more detail (more specific code, rendering context, etc.)

  **The prior entry in [[cms-0057f-prior-auth-api-presubmission-integration]] listed only 4 paStatus values (`no-auth | auth-needed | satisfied | performpa`) — the `conditional` value was omitted. This is a correction.**

- **The coverage-information system action conveys paStatus and a `satisfied-pa-id` (X12 PA reference number) via FHIR Task resource extensions.** The extension URLs are defined in the CRD IG's `StructureDefinition` profiles (CodeSystem: `http://hl7.org/fhir/us/davinci-crd/CodeSystem/temp`). cliqueue's hospital agent must parse the returned `systemActions[].resource` (a FHIR Task) and extract the coverage-information extension to read paStatus and satisfied-pa-id. The Da Vinci CRD also defines a `davinci-crd.configuration` extension for per-call payer configuration that clients can include to control which types of cards are returned. ([Da Vinci CRD STU2.1 deviations](https://hl7.org/fhir/us/davinci-crd/STU2.1/deviations.html))

- **The CRD IG mandates that all CRD server calls be invoked in parallel for multi-coverage patients.** For a patient with both primary (e.g., UHC) and secondary (e.g., Medicaid) coverage, the hospital agent must fire concurrent CDS Hooks requests to both payer CRD endpoints and handle responses independently. Coverage cards for secondary payers should be suppressed when primary already returns `satisfied`.

**Design implication:** cliqueue's TypeScript hospital agent must implement a hand-rolled CDS Hooks 2.0 client with typed interfaces for `CdsHooksRequest<TContext, TPrefetch>`, `CdsHooksResponse`, `Card`, and `SystemAction`. The client should: (1) call `GET /cds-services` on each registered payer CRD endpoint at session start to cache the prefetch template and hook service ID, (2) resolve prefetch FHIR queries against the hospital FHIR server before firing `order-sign`, (3) `POST` the hook call, (4) parse the `systemActions[].resource` Task for the coverage-information extension to extract paStatus from `coveragePaDetail` (5 valid values including `conditional`), and (5) map the result to the `paStatus` byte anchored in the on-chain `ClaimSubmitted` event. The `paStatus` byte encoding must include all 5 values: `0x00=NOT_CHECKED`, `0x01=NO_AUTH`, `0x02=AUTH_NEEDED`, `0x03=SATISFIED`, `0x04=PERFORMPA`, `0x05=CONDITIONAL`. The prior research file [[cms-0057f-prior-auth-api-presubmission-integration]] must be amended to add the `conditional` paStatus value.

**Open questions generated:**
1. Should the payer CRD discovery response (`GET /cds-services` prefetch template) be cached at hospital onboarding time or re-fetched on each claim submission — and does caching create a staleness risk if the payer updates its prefetch requirements between CMS-0057-F compliance deadline updates?
2. For the `paStatus = conditional` case (cannot determine PA requirement without more detail), should cliqueue treat this identically to `auth-needed` (pause on-chain commitment, trigger PAS) or route to human review — and should `conditional` map to a distinct `paStatus` byte value in `ClaimSubmitted` to enable auditors to distinguish it from `auth-needed`?
3. Should the hand-rolled CDS Hooks TypeScript client be published as a standalone `@cliqueue/cds-hooks-client` package under an open-source license — filling the ecosystem gap and providing hospital integration engineers a reusable typed client for both CRD and future Da Vinci IG integrations?

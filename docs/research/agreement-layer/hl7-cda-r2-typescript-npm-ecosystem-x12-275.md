## 2026-05-17 — HL7 CDA R2 TypeScript npm ecosystem for X12 275 attachment payloads: no dedicated library exists; @medplum/ccda is closest but insufficient for CMS-0053-F use; hand-roll as @cliqueue/cda-attachments confirmed third ecosystem gap

**Question answered:** Does any open-source TypeScript or Node.js npm library support HL7 CDA R2 document generation for X12 275 claims attachment payloads under CMS-0053-F — or must cliqueue hand-roll a CDA template generator?

- **No dedicated open-source TypeScript/Node.js npm library exists for HL7 CDA R2 document generation in the X12 275 / CMS-0053-F attachment context.** A broad search across npm (search: `cda hl7`), GitHub, and the Perplexity live index returned no package whose stated purpose is generating HL7 CDA R2 XML for claims attachments. No package advertises CMS-0053-F compliance, X12 275 payload packaging, or claims-attachment-specific CDA template support. ([npm CDA search](https://www.npmjs.com/search?q=CDA); [npm HL7 search](https://www.npmjs.com/search?q=keywords%3AHL7))

- **`@medplum/ccda` is the closest existing library** — an actively maintained npm package (last docs update Feb 11, 2026; part of Medplum's ONC-certified platform) providing bidirectional C-CDA ↔ FHIR conversion (`convertFhirToCcda`, `convertCcdaToFhir`, `convertCcdaToXml`, `convertXmlToCcda`). However, it is narrowly scoped to **CCD/Patient Summary export** (using IPS as the FHIR bridge), not arbitrary attachment document types (consult notes, operative notes, progress notes, lab results) required under CMS-0053-F. It does not mention X12 275, CMS-0053-F, or claims attachment context. It is not demonstrated as R2.1-conformant for payer attachment exchange. ([Medplum C-CDA docs](https://www.medplum.com/docs/integration/c-cda); [npm @medplum/ccda](https://www.npmjs.com/package/@medplum/ccda))

- **`hl7.cda.template` (v1.0.0, CC0-1.0, 16 releases, published March 5, 2026) and `hl7.cda.uv.core` are FHIR package registry artifacts** — not app-level CDA XML generators. They package CDA schema/template definitions for FHIR IG tooling, not for Node.js application use. No weekly download count is meaningful as they are consumed by IG build pipelines, not npm install chains. ([libraries.io: hl7.cda.template](https://libraries.io/npm/hl7.cda.template); [FHIR registry: hl7.cda.uv.core](https://registry.fhir.org/package/hl7.cda.uv.core%7C2.0.0-sd))

- **Java (MDHT, Eclipse EMF-based) is the dominant CDA R2 tooling ecosystem; no maintained TypeScript port exists.** MDHT (Model-Driven Health Tools) is the canonical CDA R2 generator in the Java/Eclipse ecosystem. No established npm TypeScript port of MDHT or equivalent has been published. The HL7 core CDA repo (`github.com/HL7/CDA-core-2.0`) publishes schemas and FHIR logical models as IG artifacts, not a TypeScript SDK. ([HL7/CDA-core-2.0](https://github.com/HL7/CDA-core-2.0); [HL7 CDA core IG build](https://build.fhir.org/ig/HL7/CDA-core-2.0/))

- **The recommended hand-roll approach for Node.js is `xmlbuilder2` for XML serialization** with careful namespace control (`urn:hl7-org:v3`), ordered elements, `xsi:type` attributes, and CDA header metadata (templateId, effectiveTime, code, confidentialityCode, languageCode). `fast-xml-parser` is useful for parsing incoming CDA only. Neither is a CDA authoring engine — the developer must model the CDA object structure and template constraints. (Perplexity synthesis; no primary citation; treat as weak/advisory)

- **CMS-0053-F (Federal Register 2026-05676) requires X12 275 + HL7 CDA for attachment exchange, effective May 26, 2028.** The CDA document type varies by attachment purpose — C-CDA R2.1 templates (Consult Note, Operative Note, Progress Note, etc.) identified by LOINC document type code in the CDA header. No single universal template covers all attachment types; the applicable template depends on the payer's attachment request. ([Federal Register 2026-05676](https://www.federalregister.gov/documents/2026/03/24/2026-05676/administrative-simplification-adoption-of-standards-for-health-care-claims-attachments-transactions))

- **This is a confirmed third TypeScript ecosystem gap alongside `@cliqueue/cds-hooks-client` and `@cliqueue/pa-lifecycle`.** All three require hand-rolled implementation: no upstream dependency exists for any of (a) CDS Hooks 2.0 typed client, (b) Da Vinci CRD/PAS pa-lifecycle encoding, or (c) HL7 CDA R2 attachment document generation. A `@cliqueue/cda-attachments` package built on `xmlbuilder2` should be the fourth module in the `@cliqueue/*` monorepo.

**Design implication:** `PayerAdapterInterface.ClaimsAttachmentAdapter` (the sub-interface added by the CMS-0053-F research on 2026-05-17) must be implemented by a hand-rolled `@cliqueue/cda-attachments` TypeScript module using `xmlbuilder2`. The module must support at minimum: (1) CDA R2 header generation (templateId, effectiveTime, LOINC code, confidentialityCode, author, custodian), (2) per-attachment-type section scaffolding (indexed by C-CDA R2.1 template OID), and (3) serialization to XML string for X12 275 payload. `@medplum/ccda` should be evaluated as a `devDependency` for test fixture generation (CCD round-trips) but must not be used as the primary attachment generator due to scope mismatch.

## 2026-05-17 — `@cliqueue/cda-attachments` scope confirmed: 12 C-CDA R2.1 named document templates; CMS-0053-F is claims-only (prior auth excluded); pure Node.js Schematron validation via `cda-schematron-validator` is viable; Docker CI alternative available

**Question answered:** Should `@cliqueue/cda-attachments` scope to only CMS-0053-F listed attachment types or build a generic C-CDA R2.1 template engine — and should it validate against HL7 CDA R2 schema + C-CDA Schematron rules in CI via Java or pure Node.js?

- **CMS-0053-F final rule (Federal Register 2026-05676, March 24, 2026) adopts: (1) X12N 275/277 v6020, (2) HL7 C-CDA IG Volume One, (3) HL7 C-CDA IG Volume Two, (4) HL7 CDA R2 Attachment IG (March 2022 iteration).** Compliance deadline is **May 26, 2028** for all covered claims-attachment standards — no separate deadlines by attachment category. ([CMS fact sheet](https://www.cms.gov/newsroom/fact-sheets/administrative-simplification-adoption-standards-health-care-claims-attachments-transactions); [Federal Register 2026-05676](https://www.federalregister.gov/documents/2026/03/24/2026-05676/administrative-simplification-adoption-of-standards-for-health-care-claims-attachments-transactions))

- **CMS-0053-F does NOT cover prior authorization attachments** — the rule is strictly claims attachments. PA attachment standards were proposed but not finalized in this rulemaking. This means `@cliqueue/cda-attachments` scope is limited to post-claim medical record submission; the CRD/PAS prior auth flow (already implemented in `@cliqueue/pa-lifecycle`) is governed by CMS-0057-F, not this rule. ([cubetherapybilling.com CMS-0053-F explainer](https://www.cubetherapybilling.com/cms-0053-f-explained-what-aba-providers-need-to-know-before-may-2028); [Gainwell analysis](https://www.gainwelltechnologies.com/resources/blogs/cms-0053-f-standardizing-electronic-claims-attachments/))

- **The 12 C-CDA R2.1 named document templates confirmed from the build.fhir.org IG:** Care Plan (including HHPoC), Consultation Note, Continuity of Care Document (CCD), Discharge Summary, History and Physical (H&P), Operative Note, Procedure Note, Progress Note, Referral Note, Transfer Summary, Unstructured Document, Patient Generated Document (US Realm Header). ([C-CDA 2.1 IG on build.fhir.org](https://build.fhir.org/ig/HL7/CDA-ccda-2.1-sd/); [C-CDA IG Project Life Dashboard](https://projectlifedashboard.hl7.org/specifications/hl7-cda-r2-implementation-guide-consolidated-cda-templates-for-clinical-notes-release-2-1/))

- **There is no "Diagnostic Imaging Report" or "Laboratory Report" as a standalone C-CDA document-level template.** Lab results and imaging data are represented as sections/entries within other document types (Progress Note, H&P, Consult Note). This means `@cliqueue/cda-attachments` must model lab/imaging content as section-level structures inside appropriate document templates, not as independent template types. (Confirmed from build.fhir.org C-CDA 2.1 IG — no separate Imaging Report or Lab Report document template listed.)

- **Pure Node.js Schematron validation is viable via `cda-schematron-validator` (v1.1.12, published October 23, 2022).** This is a fork of Eric Wadkins' `cda-schematron` specifically for C-CDA validation, used in HL7's own `fhir-cda-validation` tooling. It is pure Node.js using `xmldom` + `xpath` — no Java subprocess required. It is the most recently maintained option (Oct 2022 vs Feb 2017 for `cda-schematron`). The `schematron-runner` package (v1.0.0, Dec 2018) is an alternative but older and less maintained. ([npm registry: cda-schematron-validator v1.1.12](https://registry.npmjs.org/cda-schematron-validator); [GitHub: HL7/fhir-cda-validation](https://github.com/HL7/fhir-cda-validation/blob/main/README.md); [GitHub: cda-schematron-server](https://github.com/ewadkins/cda-schematron-server))

- **The original `cda-schematron` package (v1.0.1, published February 3, 2017) is stale — 9 years without a release.** It should not be used as a primary dependency; `cda-schematron-validator` is the maintained fork. ([npm registry: cda-schematron v1.0.1](https://registry.npmjs.org/cda-schematron))

- **Docker-based CI validation is a viable alternative** via the `KvalitetsIT/cda-validator` container (Java-based, runs as a local HTTP service on port 8080). This provides stricter conformance checking than the Node.js Schematron packages but adds Docker-in-CI overhead. It is not an official ONC/NIST reference tool. ([GitHub: KvalitetsIT/cda-validator](https://github.com/KvalitetsIT/cda-validator))

- **NIST CDA Validation tool is web-service only** — the public online validator was decommissioned in 2018. No NIST CLI or Docker image exists for download. ([NIST CDA validation site](https://cda-validation.nist.gov/cda-validation/))

**Design implication:** `@cliqueue/cda-attachments` should target the **6 practically relevant C-CDA R2.1 document templates for claims attachments** (Progress Note, Consultation Note, Operative Note, Procedure Note, Discharge Summary, Unstructured Document) and the CMS-0053-F content categories (medical records, imaging sections, lab sections, clinical notes) as section-level structures within those templates — not a generic 12-template engine. CI validation should use `cda-schematron-validator` (pure Node.js, no Java in CI) as the primary gate, with known-valid XML fixtures against C-CDA Schematron rules. Prior auth attachment scope belongs to `@cliqueue/pa-lifecycle` (CDS Hooks/PAS), not this package.

**Open questions generated:**
1. Should `@cliqueue/cda-attachments` include a `cdaAttachmentScope` TypeScript constant listing the 6 supported document template OIDs and their LOINC document type codes — published in the hospital onboarding checklist as a formal scope declaration, so hospital integration engineers know which attachment types trigger the CMS-0053-F path vs. the manual-submission fallback?
2. Does `cda-schematron-validator` v1.1.12 correctly validate documents against the full C-CDA R2.1 Schematron rule set (including VSAC value set assertions) — or does it require separate VSAC API access to resolve value set bindings, which would require a NLM API key in CI?
3. Should `@cliqueue/cda-attachments` publish a set of known-valid CDA XML fixtures (one per supported document type) as the primary CI validation artifact — allowing payers to verify attachment conformance against the fixtures without running the Schematron engine themselves?

# TimelockController vs Compound Timelock — UUPS Proxy Upgrade Path

## 2026-05-15 — Should `ClaimsAdjudicator` use Compound/Uniswap-style Timelock.sol or OZ's `TimelockController` with overridden `updateDelay()` — and does the choice affect the UUPS proxy upgrade path?

**Context:** Prior research ([[timelock-min-delay-immutable-vs-mutable]]) recommended extending OZ `TimelockController` with an overridden `updateDelay()` that enforces `MIN_DELAY_FLOOR = 48h`, or using Compound/Uniswap standalone `Timelock.sol`. It left open whether the choice affects the UUPS proxy upgrade path. This document answers that question.

---

### Findings

- **The UUPS `_authorizeUpgrade` hook does NOT check `msg.sender == address(this)` or any specific caller identity.** It is declared `internal virtual` with no default implementation — developers must override it with any access control of their choice. The only built-in check is the `onlyProxy` modifier on `upgradeToAndCall`, confirming the call occurs through the proxy's delegatecall context. Sources: [OpenZeppelin UUPSUpgradeable.sol (master)](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/proxy/utils/UUPSUpgradeable.sol), [RareSkills UUPS explainer](https://rareskills.io/post/uups-proxy)

- **The canonical TimelockController + UUPS pattern is: set `_authorizeUpgrade` to use `onlyOwner`, then call `transferOwnership(address(timelockController))`.** The TimelockController becomes the proxy's owner. When a governance proposal passes, the TimelockController calls `upgradeToAndCall` through its `execute()` function after the configured delay. The `_authorizeUpgrade` check passes because `msg.sender == timelockController == owner`. This pattern works identically with either OZ `TimelockController` or Compound `Timelock.sol` — both can call `upgradeToAndCall` as `msg.sender`. Source: [OZ Forum — Timelock for upgradeable contracts](https://forum.openzeppelin.com/t/timelock-for-upgradeable-contracts/42148), [OZ Forum — Gnosis Safe + Timelock upgrade tutorial](https://forum.openzeppelin.com/t/tutorial-on-using-a-gnosis-safe-multisig-with-a-timelock-to-upgrade-contracts-and-use-functions-in-a-proxy-contract/7272)

- **Compound/Uniswap `Timelock.sol` has NO OZ `AccessControl` — it uses a single-admin/pendingAdmin pattern.** There is no PROPOSER_ROLE, EXECUTOR_ROLE, or CANCELLER_ROLE. Only the admin can queue and execute transactions. OZ `TimelockController` uses full `AccessControl` with composable roles, allowing a multisig as PROPOSER and a separate automated keeper as EXECUTOR. For cliqueue's 3-of-5 multisig proposer model, OZ `TimelockController` is the more suitable base — Compound `Timelock.sol` would require its admin to be the multisig directly. Sources: [Compound Timelock.sol source](https://github.com/compound-finance/compound-protocol/blob/master/contracts/Timelock.sol), [OZ TimelockController.sol source](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/governance/TimelockController.sol)

- **Compound v3, Aave v3, and Uniswap do NOT use UUPS for their core settlement contracts.** Aave v3 uses the transparent proxy + ProxyAdmin pattern. Compound v3 uses a custom `Unitroller/CometProxyAdmin` pattern (functionally similar to UUPS). Uniswap uses transparent proxy. Their timelocks serve as the upgrade authority in all cases. This is a meaningful signal: major DeFi protocols with active governance avoid UUPS for settlement-critical contracts because UUPS upgrade-logic lives in the implementation — a misconfigured new implementation can cause an unrecoverable lockout. Sources: [Compound v3 governance docs](https://docs.compound.finance/governance/), [Aave governance](https://aave.com/docs/developers/governance), [Medium — proxy patterns comparison](https://medium.com/coinmonks/contract-upgradeability-how-to-build-protocols-that-can-evolve-without-losing-trust-c7f19ecd1bd0)

- **Known UUPS risk: unrecoverable deadlock on v4→v5 storage layout migration (OZ Issue #6362).** If a UUPS upgrade changes the storage layout in a way that moves the upgrade authorization logic, the new implementation may be unable to authorize further upgrades. OZ v5 introduced ERC-7201 namespaced storage which is incompatible with v4's sequential storage layout. This is an active, documented risk with no automatic mitigation. Source: [OZ GitHub Issue #6362](https://github.com/openzeppelin/openzeppelin-contracts/issues/6362)

- **Transparent proxy + OZ `TimelockController` (with `MIN_DELAY_FLOOR` override) is the safer production choice for `ClaimsAdjudicator`.** The ProxyAdmin holds upgrade logic independently of the implementation, eliminating the UUPS deadlock risk. OZ's `TimelockController` with AccessControl allows composable role separation (3-of-5 multisig PROPOSER, automated EXECUTOR, emergency PAUSER not behind timelock). The `updateDelay()` override enforcing `MIN_DELAY_FLOOR = 172800` seconds (48 hours) is straightforward in either pattern — it does not interact with the proxy type choice. Source: [OZ proxy README](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/proxy/README.adoc), [OZ access control docs](https://docs.openzeppelin.com/contracts/5.x/access-control)

- **Somnia EVM compatibility with Solidity `immutable` keyword: compatible but EVM hardfork level unconfirmed in public docs.** Somnia states "all Solidity features work identically" and "same gas mechanics and opcodes." Somnia's AOT (ahead-of-time) EVM compilation is consistent with Istanbul or later (where `immutable` is fully supported). No explicit hardfork level (Istanbul/London/Shanghai/Cancun) is published in Somnia developer FAQs as of May 2026. The `immutable` keyword almost certainly works on Somnia, but cliqueue's deployment runbook should confirm via a test deployment of a contract with an `immutable` variable before production. Sources: [Somnia Developer FAQs](https://docs.somnia.network/developer/deployment-and-production/support-and-community/developer-faqs), [Somnia network description](https://docs.somnia.network)

---

**Design implication:** Switch `ClaimsAdjudicator` from UUPS to **transparent proxy + OZ `TimelockController`** (with `updateDelay()` override enforcing `MIN_DELAY_FLOOR = 48h` and PROPOSER_ROLE assigned to the 3-of-5 multisig). This matches the production pattern of every major DeFi protocol with active governance, eliminates the UUPS upgrade deadlock risk, and integrates cleanly with the AccessControl role model needed for hospital-scoped admin separation.

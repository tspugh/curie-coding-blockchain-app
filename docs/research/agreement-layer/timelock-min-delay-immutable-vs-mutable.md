# TimelockController Minimum Delay ŌĆö Immutable vs. Mutable Design

## 2026-05-15 ŌĆö Should the `TimelockController` minimum delay in `ClaimsAdjudicator` be stored as a mutable parameter or hard-coded as an immutable constant?

**Context:** The prior stablecoin governance research ([[stablecoin-address-governance-timelock]]) recommended a 48-hour `TimelockController` minimum delay for `stablecoinAddress` changes in `ClaimsAdjudicator`. It left open whether the delay itself should be a mutable state variable (updatable only through the timelock ŌĆö OpenZeppelin's default) or an immutable constant (set at construction, unreducible by any governance action). This matters because a governance decision or compromised multisig could shorten the delay to zero, negating the enterprise notice-period guarantee that hospital procurement relies on.

---

### Findings

- **OpenZeppelin `TimelockController` (current, v5.x) has NO immutable floor on `minDelay`.** The `updateDelay()` function accepts any value including zero, enforcing only that `msg.sender == address(this)`. There is no `minDelayBaseline` or `MINIMUM_DELAY` constant in the standard library implementation. This is a known, documented limitation since at least 2021 (OZ Issue #2642 ŌĆö feature request for immutable baseline was filed but not merged into the standard library as of May 2026). Source: [OpenZeppelin TimelockController.sol (master)](https://github.com/OpenZeppelin/openzeppelin-contracts/blob/master/contracts/governance/TimelockController.sol), [OZ Issue #2642](https://github.com/OpenZeppelin/openzeppelin-contracts/issues/2642)

- **Compound and Uniswap use hardcoded `MINIMUM_DELAY` constants (2 days) enforced in `setDelay()`.** Both protocols define `uint public constant MINIMUM_DELAY = 2 days` and `uint public constant MAXIMUM_DELAY = 30 days` as compile-time constants. Their `setDelay()` function requires `delay_ >= MINIMUM_DELAY`, making it impossible for any governance action to reduce the delay below the floor. This is the production-battle-tested pattern for enterprise-grade timelocks. Source: [Uniswap governance/Timelock.sol](https://github.com/Uniswap/governance/blob/master/contracts/Timelock.sol), [Compound compound-protocol/Timelock.sol](https://github.com/compound-finance/compound-protocol/blob/master/contracts/Timelock.sol)

- **The "mutable but self-gated" pattern has a real exploit history.** A 2021 vulnerability in `TimelockController` allowed an actor with the executor role to reset the delay to 0 immediately and escalate privileges. OpenZeppelin patched the executor role restriction but did not add a delay floor. The Beanstalk exploit ($182M, April 2022) is the canonical case: a 0-delay emergency path allowed same-transaction governance attack. The lesson from both: any path to delay=0 is critical-severity regardless of who can trigger it. Sources: [OZ Security Advisory GHSA-fg47-3c2x-m2wr](https://github.com/OpenZeppelin/openzeppelin-contracts/security/advisories/GHSA-fg47-3c2x-m2wr), [Immunefi ŌĆö Beanstalk Governance Attack](https://medium.com/immunefi/hack-analysis-beanstalk-governance-attack-april-2022-f42788fc821e)

- **The bootstrap paradox for a mutable delay is resolvable: the timelock governs its own delay change.** If `minDelay` is a mutable parameter behind the timelock, then any reduction itself requires a 48-hour queued transaction ŌĆö the attacker cannot compress the delay to zero in one step. However, once a governance proposal shortens it to (say) 1 hour, the next step to 0 costs only 1 hour. The Compound/Uniswap pattern (immutable floor + mutable working delay) is strictly superior: it collapses the multi-step attack to a single impossible step without any bootstrap paradox, since the immutable constant is set once at deploy time and requires no governance. Source: [Coinsbench ŌĆö Locking Down Security: A Guide to TimeLocks in Solidity](https://coinsbench.com/locking-down-security-a-guide-to-timelocks-in-solidity-533d8647022c)

- **MakerDAO `DSPause` uses governance-controlled delay (not immutable), but documents the risk explicitly.** MakerDAO's GSM Pause Delay is mutable through governance (not immutable), with the protocol documentation explicitly noting: "when the pause delay is set to the minimum, it is easier to pass malicious governance actions." MakerDAO increased its delay from 12 to 72 hours following governance attack concerns. This provides no floor guarantee; it relies on community vigilance. Source: [Maker Protocol Docs ŌĆö DSPause](https://docs.makerdao.com/smart-contract-modules/governance-module/pause-detailed-documentation), [MakerDAO GSM Pause Delay](https://makerdao.world/en/learn/governance/param-gsm-pause-delay/)

- **Hospital enterprise procurement context strengthens the case for immutable floor.** Unlike DeFi protocols where governance actors are token holders who can self-protect by exiting, cliqueue's counterparties are hospitals and payers operating under multi-year contracts and procurement SLAs. A hospital legal team signing a deployment approval is relying on the notice period being a durable guarantee, not a governance variable. Making `MIN_DELAY_FLOOR` immutable converts it from a policy into a protocol-enforced guarantee ŌĆö exactly the kind of documentation hospitals need for risk committees. Source: [Halborn ŌĆö What Is a Timelock Contract?](https://www.halborn.com/blog/post/what-is-a-timelock-contract)

- **Recommended pattern for `ClaimsAdjudicator` governance:** Extend or wrap `TimelockController` with two changes: (1) add `uint256 public immutable MIN_DELAY_FLOOR` initialized at construction (recommended value: 172,800 seconds = 48 hours); (2) override `updateDelay()` with `require(newDelay >= MIN_DELAY_FLOOR, "delay below floor")`. The working `_minDelay` remains mutable (can be raised by governance for higher-stakes operations), but can never be reduced below `MIN_DELAY_FLOOR`. This mirrors the Compound/Uniswap pattern and adds no gas overhead at call time (the floor is an immutable read, not a storage slot).

---

**Design implication:** `ClaimsAdjudicator` should NOT use stock `OpenZeppelin TimelockController` directly ŌĆö it must either use the Compound/Uniswap-style timelock (with hardcoded `MINIMUM_DELAY = 48 hours`) or a thin custom wrapper that overrides `updateDelay()` to enforce an immutable floor. The current OZ TimelockController's `updateDelay()` accepts delay=0, which is unacceptable for a healthcare settlement contract where the notice period is a procurement commitment.
